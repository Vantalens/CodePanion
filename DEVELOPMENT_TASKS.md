# RemindAI 开发任务

## 产品方向

RemindAI 是一个本地优先的个人 AI 工作流中控台。

产品路线分为两个阶段：

1. **阶段 1：个人本地控制台**
   - 统一接入本机 AI 任务
   - 清晰展示当前任务状态
   - 暴露等待输入、完成和失败等关键事件
   - 让用户能从一个界面查看上下文并直接接管

2. **阶段 2：本地 AI 工作流操作台**
   - 将重复工作沉淀为可复用的本地工作流
   - 支持工作流模板、任务编排、结果历史和回放
   - 始终保持个人使用、本地优先、单用户边界

### 明确不做

- 不做团队版
- 不做多用户协作
- 不做权限、审批流、共享空间和企业管理能力
- 不试图替代 Codex、Claude Code、Copilot CLI 或 VS Code 自身的对话产品形态

---

## 当前状态

### 已具备

- 本地 daemon，提供 HTTP 与 WebSocket API
- CLI/PTTY 会话监控
- 提示检测与回复注入
- GUI 时间线与会话视图
- Codex Desktop 本地工作流同步
- VS Code 来源注册与轻量事件接入
- 外部来源注册 API

### 当前缺口

- 还没有稳定的首方自动化测试基线
- 会话与工作流数据仍以纯内存为主，缺少持久化和保留策略
- 各来源的接入质量还不均衡
- 阶段 1 的验收标准还没有完全转化成可执行检查
- 在阶段 1 尚未稳定进入日常使用前，不应启动阶段 2

---

## 开发原则

1. **先稳定，再扩展**
   - 先完成阶段 1 的可靠性，再开启阶段 2 的产品工作。

2. **优先做深，不盲目增加适配器数量**
   - 先提升来源质量、状态保真度和回复可靠性，再增加更多浅层接入。

3. **坚持本地优先**
   - 默认使用本地存储、本地 API 和清晰的适配器边界。

4. **行为变化必须同步文档**
   - 只要产品范围、验证流程或用户可见行为发生变化，就同步更新本文档、README 和相关文档。

5. **完成定义必须包含验证**
   - 任何任务只有在对应路径完成构建、测试或冒烟验证后，才能算完成。

# 阶段 1：个人本地控制台

## 阶段目标

让 RemindAI 稳定到足以每天常驻，用来同时监控并接管多个本地 AI 任务。

## 阶段 1 退出标准

- 用户可以在一个 GUI 中看到所有活跃的本地 AI 任务。
- 等待输入的任务能够被快速发现，且无需切换窗口也能识别。
- 对于支持交互的任务，用户可以从 RemindAI 中回复并让任务继续执行。
- 主要日常使用链路不会出现失控的内存增长。
- 核心路径具备稳定的自动化验证基线。
- 文档能够真实反映当前产品状态和已支持能力。

## P0：基础与可靠性

### P0.1 建立首方自动化测试基线

**目标：** 为最关键的行为建立自动化安全网。

**目标覆盖范围：**

- `promptDetector`
- `sessionManager`
- `sourceManager`
- `workflowManager`
- `codexDesktopAdapter`
- 关键 HTTP/WebSocket 行为

**交付物：**

- 根目录统一测试命令
- 核心管理器单元测试
- 关键 daemon 路由集成测试
- 本地测试流程文档

**验收标准：**

- 全新检出仓库后，可以按文档成功运行测试命令。
- 对提示、会话、来源或工作流的修改都有回归覆盖。

### P0.2 增加持久化与保留边界

**目标：** 避免长时间运行时完全依赖无限增长的纯内存状态。

**交付物：**

- 工作流线程与关键历史的持久化方案
- 会话输出、工作流条目、来源事件的保留策略
- daemon 重启后的行为定义与验证
- 临时实时状态与持久历史之间的清晰边界

**验收标准：**

- daemon 重启后，不会丢失 GUI 继续有用所需的最小历史。
- 长时间运行不会出现无限制内存增长。
- 保留策略有文档说明，并在需要处支持配置。

### P0.3 固化阶段 1 验收场景

**目标：** 把阶段 1 的产品承诺转成可重复检查。

**必须覆盖的场景：**

- 多个 CLI 会话同时运行
- 一个或多个会话等待输入
- 回复准确写回对应会话
- Codex Desktop 线程接入
- VS Code 来源注册
- 中文文本在 daemon、GUI、WebView 全链路不乱码
- daemon 重启后 GUI 可重新连接

**交付物：**

- 阶段 1 验收清单
- 手工冒烟测试指南
- 可自动化部分对应的自动化检查

**验收标准：**

- 后续开发者无需依赖口口相传，也能判断阶段 1 是否仍然成立。

---

## P1：来源质量与体验完整度

### P1.1 提升来源保真度

**目标：** 让每一个已经支持的来源都足够有用，值得出现在界面上。

**交付物：**

- 定义每类来源应达到的事件质量
- 统一规范来源元数据
- 改善 Codex Desktop 线程标题与状态质量
- 在公开 API 允许的范围内，提升 VS Code 事件价值
- 清楚记录不支持或刻意不读取的数据

**验收标准：**

- GUI 中展示的每个来源，都能帮助用户判断当前最需要关注什么。
- 文档不会把某个来源描述得比真实能力更强。

### P1.2 强化 GUI 的任务分诊能力

**目标：** 让 GUI 更快回答“现在最需要我处理什么？”

**交付物：**

- 更清晰的任务优先级状态
- 更好的活跃、等待、完成、错误分组与筛选
- 更醒目的可回复提示
- 更有用的工作流摘要

**验收标准：**

- 用户面对多个并行任务时，可以一眼判断下一步该处理谁。

### P1.3 提升重连与失败场景可靠性

**目标：** 让控制台在普通本地故障下仍然可判断、可恢复。

**交付物：**

- 清晰的重连行为
- 更好的过期来源处理
- 更明确的 daemon 启停状态
- 适配器与客户端更有用的失败日志

**验收标准：**

- 短暂的 daemon 或 GUI 中断，不会让用户无法判断真实任务状态。

## P2：阶段 1 产品收口

### P2.1 清理安装与打包流程

**目标：** 让本地安装和日常启动变得可预测。

**交付物：**

- 一个当前唯一有效的安装路径
- 准确的平台支持说明
- daemon 与 GUI 的启动行为验证
- 删除过时的发布或安装说明

**验收标准：**

- 新用户可以只看一份准确指南就完成安装和运行。

### P2.2 整理文档体系

**目标：** 让文档始终和当前产品一致，不再继续积累旧路线噪音。

**交付物：**

- README、架构、API、来源说明、安装、故障排查文档统一
- 历史报告清楚标记为历史资料
- 每完成一个阶段性里程碑后，同步更新 `DEVELOPMENT_TASKS.md`

**验收标准：**

- 文档不会夸大当前能力。
- 仓库内的产品定位始终一致。

# 阶段 2：本地 AI 工作流操作台

## 进入条件

只有同时满足以下条件，才允许启动阶段 2 的产品开发：

- 阶段 1 退出标准已经满足
- 核心自动化测试已经建立
- 持久化与保留策略已经实现
- 真实个人使用已经积累出足够证据，能识别值得产品化的重复流程

## 阶段目标

从“观察并接管本地 AI 任务”，升级到“设计、运行并复用本地 AI 工作流”。

## 阶段 2 候选工作流

### S2.1 工作流模板

- 保存常见本地工作流为模板
- 参数化常见输入
- 以更低成本重复运行

### S2.2 多步骤编排

- 将多个本地 AI 任务串成定义明确的步骤序列
- 支持依赖关系、暂停点和人工检查点
- 保留清晰可读的本地执行历史

### S2.3 跨工具协作

- 允许不同本地工具参与同一条工作流
- 追踪每一步由哪个工具产出
- 让跨工具交接保持显式、可检查

### S2.4 结果历史与回放

- 搜索过去的本地工作流
- 重新打开旧结果
- 用新输入重跑过去的流程

## 阶段 2 退出标准

- 重复工作可以被沉淀为可复用的本地工作流。
- 至少有一条多步骤工作流，可以完全在 RemindAI 中执行、检查并再次运行。
- 阶段 2 的能力不会破坏阶段 1 已建立的优点：本地优先、个人可用、可理解、可靠。

# 下一段开发切片

## 当前应立即推进

下一段工程开发应先做：

1. 建立首方自动化测试基线
2. 设计并确定持久化与保留方案
3. 把阶段 1 验收场景转成可重复执行的检查清单

## 为什么先做这些

这三项会降低后续所有变更的风险：

- 测试保护行为
- 持久化防止长时间运行失稳
- 验收场景保证产品不会偏离真正价值

如果没有这三项，继续叠加功能只会让不确定性增长得比产品价值更快。
